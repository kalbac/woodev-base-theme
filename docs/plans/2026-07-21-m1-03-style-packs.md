**Goal:** Ship all 8 Basecoat visual style packs as one standalone CSS bundle each, enqueue only the one the admin picks (`style_preset` theme_mod, default `vega`), and surface a real Basecoat `.btn` so the pack choice is actually visible and provable in a browser.

**Architecture:** A single JS source (`scripts/lib/packs-lib.mjs`) is the one place the 8 pack names live; it feeds both the CSS-entry generator and Vite's Rollup inputs. Each generated entry is byte-identical except one `@import 'basecoat-css/<pack>'` line — the pack imported *after* Tailwind and *before* our design tokens, mirroring the proven `app.css` order. On the PHP side a backed `StylePreset` enum validates the stored `style_preset` and maps it to its Vite manifest key; `Assets` enqueues exactly that bundle. The search form and the blog read-more link get Basecoat's `.btn`/`.input` classes so packs have a visible surface (they differ only in component shape, not colour — see the key finding below).

**Tech Stack:** PHP 8.1+ backed enum, WordPress `theme_mod`s + `get_search_form()` override, Vite 8 multi-entry + `@tailwindcss/vite`, Basecoat CSS 1.0.2 (exact pin), Vitest, Playwright.

**Prerequisites:** M1-02 merged (`f3f5f0a`). Docker running for integration + e2e. `node_modules` installed (basecoat-css 1.0.2 present).

**Git:** branch `feat/m1-style-packs` off `main`. Commit after every task.

## Key finding, read before starting (verified against basecoat-css 1.0.2 on disk)

The s1 gotcha called the 8 packs "standalone full **token** sets." That wording is imprecise and will mislead you. The real structure:

- `basecoat-css/<pack>` → `basecoat-base.css` + `styles/<pack>.css`.
- `basecoat-base.css` = `base/base.css` (the `:root`/`.dark` colour tokens — the **shadcn palette, shared by all 8 packs**) + `basecoat-components.css` (component structure).
- `styles/<pack>.css` (~1386 lines) = the pack's component **skin**: `@apply` rules that set radius, height, density. **It contains zero colour tokens** — verified: `grep -c ':root|--background:|--primary:'` is 0 for all 8.

**Consequence:** the packs differ only in **shape**, not colour (e.g. `.btn` default height is `h-9`/36px in `vega` but `h-8`/32px in `nova`; radii differ too). A pack switch is therefore **invisible on a page that renders no Basecoat component classes** — and M1-02's templates render none. That is exactly why Task 5 surfaces a `.btn`: without it, the 8 bundles would build byte-different yet look identical, and e2e could only prove filenames. Task 7 corrects the gotcha wording.

The **directive is unchanged**: one standalone bundle per pack, exactly one enqueued. Each `styles/<pack>.css` fully re-declares every component, so importing two packs doubles every component rule — never combine them.

`scripts/lib/packs-lib.mjs`:

```js
// Single source of truth for the 8 Basecoat style packs (spec §6).
// Consumed by scripts/build-pack-entries.mjs (emits one CSS entry per pack) and
// vite.config.mjs (one Rollup input per pack). Basecoat forbids combining packs —
// each is a standalone bundle, exactly one enqueued at runtime
// (docs/gotchas/basecoat-style-packs-standalone.md).
export const PACKS = ['vega', 'nova', 'maia', 'lyra', 'mira', 'luma', 'sera', 'rhea'];

// Basecoat's own default pack, and therefore the theme's default when the
// style_preset theme_mod is unset or holds a value not in PACKS.
export const DEFAULT_PACK = 'vega';

// The CSS for one pack's standalone Vite entry. Byte-identical between packs
// except the single `@import 'basecoat-css/<pack>'` line. The import ORDER is
// load-bearing and mirrors the proven app.css structure:
//   tailwind -> pack -> our tokens -> adapter -> states.
// The pack must land AFTER tailwind and BEFORE our tokens: both Basecoat's and
// our tokens are un-layered :root rules, so source order alone decides the
// winner — moving tokens above the pack silently discards every design token
// (docs/gotchas/basecoat-tokens-are-un-layered.md). Paths are relative to the
// generated file's home, src/css/packs/.
export function packEntryCss(pack) {
  if (!PACKS.includes(pack)) {
    throw new Error(`Unknown Basecoat style pack: ${pack}`);
  }
  return `/* GENERATED by scripts/build-pack-entries.mjs — do not edit by hand.
 * Standalone CSS bundle for the Basecoat "${pack}" style pack (spec §6).
 * Import order is the contract; see the generator and
 * docs/gotchas/basecoat-tokens-are-un-layered.md.
 */
@source "../../../woodev-base-theme/**/*.php";

@layer theme, base, components, adapter, utilities;

@import 'tailwindcss';

/* The style pack. Exactly one per bundle — Basecoat forbids combining packs.
 * Only this line differs between the 8 generated entries. */
@import 'basecoat-css/${pack}';

/* MUST follow the pack: un-layered tokens win by source order. */
@import '../tokens.generated.css';

@import '../adapter/index.css' layer(adapter);
@import '../states.css';
`;
}
```

```php
<?php
/**
 * The Basecoat style pack selected for the front end.
 *
 * @package Woodev\Theme\Base
 */

declare(strict_types=1);

namespace Woodev\Theme\Base;

/**
 * The eight standalone Basecoat visual style packs (spec §6).
 *
 * Basecoat forbids combining packs, so the build emits one standalone CSS bundle
 * per case and Assets enqueues exactly the one this resolves to. M1-04 adds the
 * Customizer control that writes `style_preset`; this enum is the single place
 * that validates the stored value and maps it to a Vite build entry.
 */
enum StylePreset: string {

	case Vega = 'vega';
	case Nova = 'nova';
	case Maia = 'maia';
	case Lyra = 'lyra';
	case Mira = 'mira';
	case Luma = 'luma';
	case Sera = 'sera';
	case Rhea = 'rhea';

	/**
	 * Basecoat's default pack — the theme's default when the setting is unset or
	 * holds a value that is no longer valid.
	 */
	public static function default(): self {
		return self::Vega;
	}

	/**
	 * Resolve the admin-chosen pack from its theme_mod, validated.
	 */
	public static function fromThemeMod(): self {
		$stored = (string) get_theme_mod( 'style_preset', self::default()->value );

		return self::tryFrom( $stored ) ?? self::default();
	}

	/**
	 * The Vite manifest key (and dev-server path) for this pack's CSS bundle.
	 */
	public function cssEntry(): string {
		return "src/css/packs/{$this->value}.css";
	}
}
```

### Task 6: e2e — the right bundle loads and the pack changes the button

**Files:**
- Create: `tests/e2e/style-packs.spec.mjs`

Assert two things per pack: the enqueued stylesheet is that pack's bundle (proves the enqueue wiring), and the `.btn` has that pack's height (proves the bundles are genuinely different, not just differently named). Vega `.btn` default height is `h-9` = 36px; nova is `h-8` = 32px.

- [ ] **Step 1: Write the spec**

`tests/e2e/style-packs.spec.mjs`:

```js
// tests/e2e/style-packs.spec.mjs
//
// Guards M1-03: Assets enqueues exactly the bundle the style_preset theme_mod
// selects, and the 8 packs are genuinely different (they differ in component
// SHAPE, not colour — so we assert a .btn's height, not a colour). The blog
// index renders a "Read more" .btn per post (seeded by global-setup).
//
// Runs serially and resets the mod in finally: the mod is global to the shared
// dev site, and other specs assume the default (vega).
import { execSync } from 'node:child_process';
import { expect, test } from '@playwright/test';

/** Run a wp-cli command in the cli container, return trimmed stdout. */
function wp(command) {
  return execSync(`npx wp-env run cli wp ${command}`, {
    encoding: 'utf8',
    stdio: ['ignore', 'pipe', 'pipe'],
  }).trim();
}

/** Height (px, rounded) of the first Basecoat .btn on the current page. */
async function firstButtonHeight(page) {
  const btn = page.locator('.btn').first();
  await expect(btn).toBeVisible();
  return Math.round((await btn.boundingBox()).height);
}

test.describe.serial('Basecoat style packs', () => {
  test.afterAll(() => {
    wp('theme mod remove style_preset');
  });

  test('default pack (vega): vega bundle loads and .btn is 36px tall', async ({ page }) => {
    wp('theme mod remove style_preset');
    await page.goto('/');

    await expect(page.locator('link[rel="stylesheet"][href*="style-vega-"]')).toHaveCount(1);
    await expect(page.locator('link[rel="stylesheet"][href*="style-nova-"]')).toHaveCount(0);

    expect(await firstButtonHeight(page)).toBe(36);
  });

  test('switching to nova loads the nova bundle and shrinks the .btn to 32px', async ({ page }) => {
    wp('theme mod set style_preset nova');
    await page.goto('/');

    await expect(page.locator('link[rel="stylesheet"][href*="style-nova-"]')).toHaveCount(1);
    await expect(page.locator('link[rel="stylesheet"][href*="style-vega-"]')).toHaveCount(0);

    expect(await firstButtonHeight(page)).toBe(32);
  });
});
```

## Self-review against the spec

**Coverage.** §6 `style_preset` (8 values, default vega): the enum (Task 3) + pack-aware enqueue (Task 4). "One standalone bundle per pack, one enqueued": the generator + Vite inputs (Tasks 1–2), proven by the manifest check and e2e (Tasks 2, 6). "Scheme + primary preset apply on top": untouched here by design — they layer over whichever pack is active and are M1-04/M1-05. §7 button/form-control components: surfaced minimally (Task 5) as agreed. §5 layer order and the un-layered-token cascade: preserved per pack and sentinel-checked (Task 2 Step 5).

**Deliberately deferred** (restated so a reviewer does not read it as a gap): the Customizer control writing `style_preset` (M1-04), `color_scheme`/`primary_preset` (M1-04/M1-05), and the rest of the §7 component set (card, badge, alert, tabs, accordion, dialog) — a later adapter pass where they can be styled and demonstrated together.

**Known risk.** The e2e flips a site-global theme_mod; it runs `test.describe.serial` and resets in `afterAll`, but a crash between set and reset leaves the dev site on nova. That only affects local reruns (CI is fresh each run) and any spec asserting a specific pack — currently only this one. If a future spec asserts vega implicitly, make this reset more defensive. The exact `36`/`32` px assertions are pinned to Basecoat 1.0.2's `.btn` heights; a Basecoat bump (deliberate, per the pin) requires re-reading those values — that is the intended tripwire, not brittleness.
